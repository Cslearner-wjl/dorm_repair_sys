# 宿舍报修管理系统详细开发流程

## 1. 开发目标

本项目需要完成一个功能完整、可正常运行、数据可持久化的小型 Web 业务系统。最终系统应支持学生提交报修、管理员审核派单、维修人员处理、学生确认评价、管理员统计管理等完整流程，并能通过课程验收和报告评审。

## 2. 总体开发路线

项目采用“先闭环、后优化”的开发方式：

1. 先完成登录注册、三类角色权限、报修主流程和数据库持久化。
2. 再补充统计、用户管理、软删除、异常处理和页面细节。
3. 最后整理测试截图、课程报告、部署说明和答辩演示材料。

核心原则：

- 每个功能必须有页面、接口、数据库记录和测试结果。
- 状态流转统一由后端校验，前端只负责展示可操作按钮。
- 所有演示数据必须来自数据库，不能只写死在前端页面。
- 每天结束前记录已完成内容、未完成问题和次日计划。

## 5. 阶段二：系统设计

### 5.1 架构设计

系统采用前后端分离架构：

```text
浏览器 Vue 前端
    |
    | HTTP/JSON
    v
Spring Boot + Spring MVC + SSM REST API
    |
    | MyBatis
    v
MySQL 数据库
```

前端负责页面展示、路由权限、表单校验和接口调用。后端负责登录鉴权、角色校验、业务规则、状态流转和数据库访问。数据库负责保存用户、报修单、处理记录和评价数据。

### 5.2 数据库设计

核心表：

- `users`：用户信息、角色、账号状态、软删除时间。
- `repair_orders`：报修单基础信息、维修人员、状态、驳回原因、完成时间、软删除时间。
- `repair_records`：报修单处理记录，形成详情页时间线。
- `repair_feedbacks`：学生评价和满意度评分。

设计要点：

- 用户密码只保存加密后的哈希值。
- 报修单状态使用枚举值，避免前后端状态文案不一致。
- 派单、处理、确认等关键操作必须写入 `repair_records`。
- 删除用户和报修单优先使用软删除，避免破坏历史记录。

### 5.3 接口设计

接口统一以 `/api` 开头，统一返回：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

接口分组：

- `auth`：注册、登录、当前用户信息。
- `repairs`：报修单新增、查询、详情、修改、删除、审核、派单、处理、确认、评价。
- `users`：用户列表、维修人员列表、用户编辑、角色修改、启用禁用、软删除。
- `statistics`：总览统计、状态统计、类型统计、维修人员统计。
- `uploads`：报修图片上传，作为扩展功能。

### 5.4 输出

- 数据库表结构。
- 接口清单。
- 页面跳转关系。
- 状态流转规则。

## 6. 阶段三：后端开发

### 6.1 项目初始化

建议目录：

```text
server/
  pom.xml
  src/main/java/com/dorm/repair/
    DormRepairApplication.java
    controller/
    service/
    mapper/
    entity/
    dto/
    vo/
    config/
    interceptor/
    common/
    util/
  src/main/resources/
    application.yml
    mapper/
  uploads/
```

具体步骤：

1. 使用 Spring Initializr 或 IDE 创建 Spring Boot 项目，按 SSM 分层组织代码。
2. 添加 Spring Web、MyBatis、MySQL Driver、JWT、Lombok、Swagger / OpenAPI 等依赖。
3. 在 `application.yml` 中配置数据库连接、服务端口、JWT 密钥和可选上传目录。
4. 编写 Entity、Mapper 和 XML 映射文件，完成数据库访问。
5. 添加统一响应结构、全局异常处理和参数校验。
6. 添加 CORS 配置、日志配置、JWT 拦截器和角色权限校验。

### 6.2 登录与权限模块

开发顺序：

1. 实现用户注册接口。
2. 实现密码加密和密码校验。
3. 实现用户登录接口，返回 JWT。
4. 实现 JWT 解析拦截器或过滤器。
5. 实现角色权限拦截器。
6. 实现获取当前用户信息接口。

验收标准：

- 正确账号密码可登录。
- 错误密码返回明确提示。
- 未登录访问业务接口被拦截。
- 学生访问管理员接口被拒绝。

### 6.3 报修模块

开发顺序：

1. 学生新增报修单，默认状态为“待受理”。
2. 学生查询个人报修列表和详情。
3. 学生修改未受理报修信息。
4. 学生撤销待受理报修。
5. 管理员查询全部报修单。
6. 管理员受理或驳回报修。
7. 管理员分配维修人员。
8. 管理员软删除无效或测试报修单。

关键规则：

- 只有报修创建人可以修改或撤销自己的待受理报修。
- 已派单、处理中、待确认、已完成的报修不能被学生修改。
- 管理员驳回时必须填写驳回原因。
- 派单时必须选择维修人员。

### 6.4 维修处理模块

开发顺序：

1. 维修人员查询分配给自己的工单。
2. 维修人员将工单状态改为“处理中”。
3. 维修人员填写处理说明并提交完成。
4. 无法处理时提交原因，状态变为“需重派”。
5. 每次操作写入处理记录。

关键规则：

- 维修人员只能操作分配给自己的工单。
- 未派单工单不能开始处理。
- 已完成工单不能再次处理。

### 6.5 评价与统计模块

开发顺序：

1. 学生确认“待确认”工单。
2. 学生提交评分和评价。
3. 管理员查看统计概览。
4. 管理员查看按状态、类型、维修人员维度的统计。

关键规则：

- 只有报修创建人可以确认完成和评价。
- 评分范围为 1 到 5 分。
- 统计数据必须从数据库实时计算或查询，不能写死。

## 7. 阶段四：前端开发

### 7.1 项目初始化

建议目录：

```text
src/
  api/
  router/
  stores/
  views/
  components/
  utils/
```

具体步骤：

1. 使用 Vite 创建 Vue 3 项目。
2. 安装 Element Plus、Axios、Pinia、Vue Router。
3. 配置 Axios 基础地址、请求拦截器和响应拦截器。
4. 配置路由守卫，未登录跳转登录页。
5. 根据用户角色动态显示菜单。

### 7.2 登录注册页面

功能点：

- 登录表单。
- 注册表单。
- 手机号、账号、密码必填校验。
- 登录成功后保存 Token 和用户信息。
- 根据角色跳转到对应首页。

### 7.3 学生端页面

页面：

- 学生首页：个人报修统计、最近报修、待确认提醒。
- 新增报修页：报修类型、宿舍楼、房间号、标题、描述、联系电话。
- 我的报修页：状态筛选、类型筛选、时间筛选、详情入口。
- 报修详情页：基础信息、状态、处理记录、确认完成、评价。

关键交互：

- 待受理报修可撤销或修改。
- 待确认报修可确认完成并评价。
- 其他状态只展示详情，隐藏不可用操作。

### 7.4 管理员页面

页面：

- 管理员首页：统计卡片、类型统计、维修人员负载。
- 报修管理页：全部报修列表、筛选、审核、驳回、派单。
- 用户管理页：用户列表、角色修改、启用禁用、软删除。
- 报修详情页：处理记录、状态、当前可操作按钮。

关键交互：

- 待受理工单显示“受理”和“驳回”。
- 已受理工单显示“派单”。
- 需重派工单显示“重新派单”。
- 已完成工单只允许查看，不允许重复派单。

### 7.5 维修人员页面

页面：

- 我的任务页：待处理、处理中、待确认、已完成任务。
- 任务详情页：报修信息、联系方式、处理记录、处理说明表单。

关键交互：

- 已派单工单可开始处理。
- 处理中工单可提交完成或申请重派。
- 已完成工单只读。

## 8. 阶段五：前后端联调

### 8.1 联调顺序

1. 登录注册联调。
2. 当前用户信息和路由权限联调。
3. 学生新增报修联调。
4. 学生查询我的报修联调。
5. 管理员审核和派单联调。
6. 维修人员处理工单联调。
7. 学生确认评价联调。
8. 管理员统计和用户管理联调。

### 8.2 联调检查项

- 前端字段名是否与后端 DTO 一致。
- 时间格式是否统一。
- 状态枚举是否统一。
- 分页字段是否统一。
- 401 登录失效是否自动跳转登录页。
- 403 无权限是否展示明确提示。
- 表单错误是否能在页面上看到。

### 8.3 演示主流程

1. 使用学生账号登录。
2. 提交一条宿舍报修。
3. 切换管理员账号，受理并派单。
4. 切换维修人员账号，开始处理并提交完成。
5. 切换学生账号，确认完成并评价。
6. 切换管理员账号，查看统计结果和处理记录。

## 9. 阶段六：测试与优化

### 9.1 功能测试

| 编号 | 测试项 | 检查结果 |

- 未登录用户访问业务页面，应跳转登录页。

- 必填项为空时不能提交。
- 手机号格式错误时提示错误。
- 不存在的数据 ID 返回“记录不存在”。
- 已完成工单不能再次派单。
- 已驳回工单不能开始维修。
- Token 过期后重新登录。

### 9.4 优化内容

- 统一状态颜色和文案。
- 增加加载中、空数据、错误提示状态。
- 列表增加分页和筛选。
- 表单提交后防止重复点击。
- 页面按钮根据状态和角色动态展示。

建议使用 Git：

- `main`：稳定可演示版本。
- `dev`：日常集成分支。
- `feature/auth`：登录权限。
- `feature/repair`：报修流程。
- `feature/admin`：管理员功能。
- `feature/frontend`：前端页面。

提交信息建议格式：

```text
feat: 实现学生提交报修接口
fix: 修复管理员派单权限判断
docs: 补充数据库设计说明
test: 增加报修状态流转测试
```

## 13. 风险与应对

| 风险 | 表现 | 应对措施 |
| --- | --- | --- |
| 范围过大 | 核心流程未完成却在做扩展功能 | 先完成必做功能，图片上传和导出作为扩展 |
| 状态流转混乱 | 前端能操作不该操作的按钮 | 后端集中校验状态，前端根据后端状态展示按钮 |
| 权限漏洞 | 学生能访问管理员数据 | 所有业务接口都经过 JWT 拦截器和角色权限校验 |
| 联调困难 | 前后端字段不一致 | 先固定接口文档，再开发页面和接口 |
| 报告材料不足 | 最后缺少截图和测试记录 | 开发过程中同步保存截图和测试结果 |
| 演示数据不稳定 | 演示时没有合适数据 | 准备初始化 SQL 和固定演示账号 |

## 14. 最终验收清单

- 系统能启动，前端和后端能正常访问。
- 学生、管理员、维修人员账号均可登录。
- 学生能提交、查询、撤销、确认和评价报修。
- 管理员能审核、驳回、派单、查看统计、管理用户。
- 维修人员能查看任务、开始处理、提交完成、申请重派。
- 数据写入 MySQL，刷新页面后数据仍存在。
- 学生不能访问管理员页面和接口。
- 表单必填项、错误提示和非法访问拦截正常。
- 报修单支持新增、查询、更新和软删除。
- 页面简洁清晰，核心操作按钮明确。
- 课程报告结构完整，字数不少于 3000 中文字。
- 团队分工、工作量占比和贡献说明已填写。
- 演示账号、部署说明和初始化数据已准备完成。
