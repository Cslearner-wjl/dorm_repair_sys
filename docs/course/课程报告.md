## 摘要

随着高校宿舍规模和住宿人数不断增加，宿舍日常维修事项在登记、派单、处理和反馈等环节中容易出现信息分散、进度不透明、责任边界不清和统计困难等问题。传统线下登记或即时通讯工具沟通虽然实现成本较低，但难以形成规范的业务闭环，也不利于后续的数据追溯和管理优化。针对上述问题，本项目设计并实现了一套面向校园宿舍场景的小型 Web 业务系统，即宿舍报修管理系统。

系统采用前后端分离架构，前端使用 React 19、TypeScript 和 Vite 构建单页应用，后端使用 Java 17、Spring Boot 3、Spring MVC、MyBatis 和 MySQL 实现业务接口与数据持久化。系统围绕学生、管理员和维修人员三类角色展开，支持学生注册登录、提交报修、查看进度、确认完成和评价；支持管理员审核报修、驳回、派单、用户管理和统计分析；支持维修人员查看任务、开始处理、提交完成和申请重派。系统使用 JWT 实现登录鉴权和角色权限控制，使用统一响应结构和全局异常处理提升前后端联调一致性，使用处理记录表保存工单状态变化过程，保证业务过程可追溯。

在开发流程上，本项目按照软件工程实践要求，从需求分析、技术选型、总体架构设计、模块设计、数据库设计、接口设计、编码实现、测试验证和部署说明等环节逐步推进。测试方面，项目保存了接口测试结果，并补充了 11 个 JUnit/Mockito 单元测试，覆盖公开注册权限、角色限制、工单状态流转和手机号校验等关键规则。经构建验证，后端 `mvn test` 执行成功，前端 `npm run build` 执行成功，系统具备课程任务书要求的用户入口、基础业务功能、数据持久化、权限区分、异常处理、模块化代码结构和较完整的文档材料。

## 1. 项目背景与意义

### 1.1 项目背景

高校宿舍是学生日常学习和生活的重要空间。宿舍中的水电、门窗、家具、网络、空调和公共设施等问题具有发生频率高、处理时效要求强、涉及角色较多的特点。在线下管理模式下，学生通常通过电话、纸质登记或聊天软件向宿管人员反馈问题，管理员再通过人工方式联系维修人员。该方式在小规模场景下可以使用，但在实际运行中存在明显不足。

第一，报修信息容易分散。不同学生可能通过不同渠道提交报修信息，管理员需要在多个沟通记录之间查找，容易遗漏关键字段，例如宿舍楼、房间号、联系方式和故障描述。第二，处理进度不透明。学生提交问题后难以及时了解是否已受理、是否已派单、维修人员是否已处理，容易反复催办。第三，责任链条不清晰。管理员、维修人员和学生之间缺少统一的状态记录，出现问题时难以还原处理过程。第四，统计分析困难。学校后勤部门如果希望了解某类故障的高发区域、维修人员工作量和服务满意度，纯人工记录难以支撑快速统计。

基于上述背景，本项目选择“宿舍报修管理系统”作为课程实践题目。该选题贴近校园生活，业务规模适中，适合在课程周期内完成，同时能够覆盖 Web 系统开发中的用户认证、角色权限、业务流转、数据持久化、前后端交互、异常处理和测试验证等关键知识点，符合《软件系统设计与开发实践》课程任务书对小型 Web 业务系统的要求。

### 1.2 项目意义

从实际应用角度看，宿舍报修管理系统能够将报修业务从线下沟通转变为线上流程。学生可以在系统中统一提交报修申请并查看处理进度，管理员可以集中审核、派单和统计，维修人员可以查看自己的任务并反馈处理结果。系统不仅提升了报修效率，也增强了过程透明度和数据可追溯性。

从课程实践角度看，本项目综合运用了软件工程开发流程和主流 Web 开发技术。项目在需求分析阶段明确用户角色和业务边界，在设计阶段进行状态流转、数据库表结构和接口规范设计，在编码阶段按前后端分离和后端分层架构实现，在测试阶段进行接口测试、权限测试、异常测试和自动化测试。通过完整实践，可以加深对需求、设计、实现、测试和部署之间关系的理解，提升团队协作和工程落地能力。

### 2.2 用户角色分析

系统共设计三类用户。

| 角色 | 主要职责 | 权限范围 |
| --- | --- | --- |
| 学生 | 提交宿舍报修并反馈处理结果 | 注册登录、提交报修、查看个人工单、撤销待受理工单、确认完成、提交评价 |
| 管理员 | 负责审核、派单和系统管理 | 查看全部工单、审核报修、驳回、派单、管理用户、查看统计数据 |
| 维修人员 | 负责处理被分配的维修任务 | 查看个人任务、开始处理、提交完成结果、申请重派 |

公开注册仅允许创建学生账号。管理员账号由系统初始化产生，维修人员账号由管理员后台创建或通过角色修改产生。该设计避免普通用户通过注册接口直接获得高权限角色，符合基础权限安全要求。

### 2.3 功能性需求

系统功能围绕报修业务闭环展开。

1. 用户认证功能  
   系统支持学生公开注册、三类用户登录和当前用户信息获取。登录成功后后端返回 JWT，前端保存 token 并在后续请求中自动携带。系统根据用户角色跳转到对应首页，未登录用户访问业务接口时返回未授权提示。

2. 学生报修功能  
   学生可以填写宿舍楼、房间号、报修类型、标题、描述、联系电话和图片地址等信息并提交报修。新增报修单默认状态为待受理，系统自动生成工单号，并写入处理记录。

3. 工单查询与详情功能  
   学生可以查看自己提交的报修单，管理员可以查看全部报修单，维修人员可以查看分配给自己的任务。工单详情包含基础信息、当前状态、处理记录时间线和评价信息。

4. 管理员审核与派单功能  
   管理员可以对待受理工单进行审核，通过后进入已受理状态，也可以填写原因驳回。已受理或需要重派的工单可以分配给指定维修人员。

5. 维修处理功能  
   维修人员只能处理分配给自己的工单。工单被派发后，维修人员可以开始处理，将状态改为处理中；处理完成后填写说明并提交，状态进入待确认；如无法处理，可以申请重派。

6. 完成确认与评价功能  
   学生对待确认工单进行确认，确认后状态变为已完成。已完成工单允许学生提交 1 到 5 分评分和文字评价，每张工单只能评价一次。

7. 用户管理功能  
   管理员可以创建用户、查询用户列表、筛选维修人员、修改基础信息、修改角色、启用或禁用用户、软删除用户。

8. 统计分析功能  
   管理员可以查看报修总数、待受理数量、处理中数量、已完成数量、平均评分、状态分布、分类分布、楼栋分布、最近 N 天趋势和维修人员统计。

9. 图片上传功能  
   系统实现 `/api/uploads/images` 上传接口，支持 jpg、jpeg、png、gif、webp 等图片格式，默认最大 5MB。上传返回的 URL 可写入报修单 `imageUrls` 字段。

### 2.5 业务流程

系统核心业务流程如下。

```mermaid
flowchart LR
    A["学生登录"] --> B["提交报修"]
    B --> C["管理员审核"]
    C -->|通过| D["管理员派单"]
    C -->|驳回| H["工单结束"]
    D --> E["维修人员处理"]
    E --> F["提交完成"]
    F --> G["学生确认与评价"]
    G --> I["管理员查看统计"]
```

工单状态流转如下。

| 状态 | 说明 | 下一状态 |
| --- | --- | --- |
| PENDING | 待受理，学生刚提交 | APPROVED、REJECTED、CANCELLED |
| APPROVED | 管理员审核通过 | ASSIGNED |
| ASSIGNED | 已分配维修人员 | PROCESSING、NEED_REASSIGN |
| PROCESSING | 维修人员处理中 | WAIT_CONFIRM、NEED_REASSIGN |
| WAIT_CONFIRM | 维修人员已提交完成 | COMPLETED |
| COMPLETED | 学生确认完成并可评价 | 无 |
| REJECTED | 管理员驳回 | 无 |
| CANCELLED | 学生撤销 | 无 |
| NEED_REASSIGN | 维修人员申请重派 | ASSIGNED |

### 3.2 前端技术栈选择总结

本项目前端采用 React 19、TypeScript、Vite、Axios、lucide-react 和自定义 CSS。该技术组合适合前后端分离的业务管理系统，能够兼顾开发效率、页面交互和代码维护。

React 具有较好的组件化能力，适合将登录、学生首页、报修列表、工单详情、管理员统计、派单管理和用户管理等页面拆分为独立组件，同时复用状态标签、弹窗、分页、筛选栏、时间线等通用组件。TypeScript 提供静态类型检查，可以对用户信息、报修单、分页结果、表单参数和状态枚举进行约束，减少前后端字段不一致带来的错误。

Vite 启动快、配置简单，适合课程项目快速开发和调试；Axios 便于统一封装 REST API 请求、JWT token 和错误处理；lucide-react 提供轻量图标，能够提升按钮、分类和导航的识别度。样式方面采用自定义 CSS，而不是大型 UI 框架，主要是因为本系统页面规模适中，自定义样式可以满足布局、表单、弹窗、状态展示和响应式适配需求，也能让界面更贴合宿舍报修场景。

总体来看，这套前端技术栈与系统复杂度匹配，既能支撑多角色、多页面和多状态交互，也能保持项目结构清晰，便于课程展示和后续维护。

### 3.3 可行性分析

从技术可行性看，Spring Boot、React 和 MySQL 都是成熟稳定的 Web 开发技术，资料丰富，适合课程实践。后端采用 Spring MVC 接收请求，Service 层处理业务规则，MyBatis 操作数据库，能够清晰体现 SSM 分层思想。前端采用 React 组件化开发，页面、组件、API 封装和工具函数分离，便于维护和扩展。

从业务可行性看，宿舍报修场景角色明确、流程清晰、数据结构适中，既能覆盖课程要求的增删改查，又不会扩展成过大的业务系统。学生、管理员和维修人员之间的业务流转天然适合展示角色权限、状态机和处理记录。

从时间可行性看，项目采用“先闭环、后优化”的策略，优先实现登录、报修、审核、派单、处理、确认、评价这条主流程，再补充统计、用户管理、软删除、图片上传、测试和文档。该方式降低了范围失控风险，保证课程验收时核心功能可演示。

## 4. 系统总体架构设计

### 4.1 总体架构

系统采用前后端分离架构。浏览器运行 React 单页应用，通过 HTTP/JSON 请求访问后端 REST API。后端负责鉴权、业务规则、状态流转和数据访问，MySQL 负责保存持久化数据。

```mermaid
flowchart TB
    Browser["浏览器 / React SPA"] --> Axios["Axios API 封装"]
    Axios --> Controller["Spring MVC Controller"]
    Controller --> Service["Service 业务层"]
    Service --> Mapper["MyBatis Mapper"]
    Mapper --> DB[(MySQL 数据库)]
    Controller --> Upload["上传目录 /uploads"]
```

### 4.2 后端分层设计

后端源码位于 `server/src/main/java/com/dorm/repair/`，主要分层如下。

| 包名 | 职责 |
| --- | --- |
| `controller` | 提供 REST API，接收请求并返回统一响应 |
| `service`、`service/impl` | 定义并实现业务逻辑，集中处理状态流转和权限规则 |
| `mapper` | MyBatis Mapper 接口，连接 XML SQL 映射文件 |
| `entity` | 数据库实体对象 |
| `dto` | 请求参数对象，配合 Bean Validation 做参数校验 |
| `vo` | 响应视图对象，向前端返回展示所需字段 |
| `security`、`interceptor` | 当前用户上下文、JWT 拦截和角色校验 |
| `common` | 统一响应、错误码、业务异常、分页结果和枚举 |
| `config` | Web 跨域、上传静态资源和拦截器配置 |
| `util` | JWT 工具类等通用工具 |

### 4.3 前端结构设计

前端源码位于 `frontend/src/`，核心结构如下。

| 目录或文件 | 职责 |
| --- | --- |
| `App.tsx` | 全局登录态、路由和角色入口控制 |
| `pages/` | 页面级组件，包括登录、学生首页、报修列表、详情、管理员首页、派单、用户管理等 |
| `components/` | 通用组件，包括布局、弹窗、筛选栏、分页、状态标签、时间线、图表等 |
| `hooks/useAsyncData.ts` | 异步数据加载 Hook，封装 loading、error、reload 状态 |
| `api.ts` | Axios 客户端、token 工具和所有接口封装 |
| `types.ts` | 前后端共享的 TypeScript 类型 |
| `constants.ts` | 状态、角色、分类、演示账号等常量 |
| `utils.ts` | 日期格式化、状态文案、角色首页、统计工具等 |
| `styles.css` | 全局自定义样式 |

### 4.4 安全架构

系统使用 JWT 作为前后端分离场景下的登录凭证。用户登录成功后，后端生成 token，前端保存到本地并在请求头中携带 `Authorization: Bearer <token>`。后端拦截器解析 token，查询用户状态，并将当前用户写入 `UserContext`。带有 `@RequireRole` 的接口会校验当前用户角色，防止学生访问管理员接口或维修人员操作未分配工单。

为了避免公开注册越权，注册接口只允许创建学生账号。若请求体传入 `role=ADMIN` 或 `role=REPAIR`，后端返回业务错误；未传角色时后端强制设置为 `STUDENT`。该设计把高权限账号创建能力收敛到管理员用户管理模块中。

## 5. 功能模块设计

### 5.1 登录认证模块

登录认证模块包含学生注册、用户登录、获取当前用户和退出登录等功能。注册接口校验用户名、密码、真实姓名和手机号，并限制公开注册只能创建学生账号。登录接口校验账号是否存在、是否被禁用、密码是否正确，并返回 JWT 和用户信息。前端根据用户角色跳转到对应首页。

### 5.2 学生报修模块

学生报修模块面向普通学生用户。学生登录后可以进入提交报修页面，填写宿舍楼、房间号、分类、标题、描述、图片地址和联系电话。后端创建报修单时自动生成工单号，并将状态设置为 `PENDING`。学生还可以查看个人报修列表、筛选状态、查看详情、撤销待受理报修、确认待确认工单和提交评价。

### 5.3 管理员模块

管理员模块包括工单管理、派单调度、统计分析和用户管理。管理员可以查看全部工单，对待受理工单进行审核或驳回；对已受理和需要重派的工单分配维修人员；查看系统统计数据，包括总工单数、待受理数量、处理中数量、已完成数量、分类统计和维修人员评分。用户管理功能支持管理员创建维修人员或内部管理账号，修改用户角色、启用禁用和软删除。

### 5.4 维修人员模块

维修人员登录后只能查看分配给自己的工单。对于已派单工单，维修人员可以点击开始处理，将状态变为 `PROCESSING`；维修完成后提交处理说明，状态变为 `WAIT_CONFIRM`；如当前维修人员无法处理，可以填写原因申请重派，状态变为 `NEED_REASSIGN`，由管理员重新派单。

### 5.5 统计与可视化模块

统计模块由后端从数据库聚合计算，前端通过统计卡片、趋势图、分布图和维修人员列表展示。管理员可以了解当前工单规模、待处理压力、完成情况、平均满意度和维修人员工作量。该模块用于支撑后勤管理中的巡检安排和人员调度决策。

### 5.6 图片上传模块

图片上传模块作为系统扩展功能实现。登录用户可以调用 `/api/uploads/images` 上传图片。后端校验文件非空、大小不超过配置限制、文件类型为图片，并将文件保存到 `uploads/yyyyMMdd/` 目录。返回的 URL 可写入报修单或维修完成说明中，便于展示现场问题和维修结果。

## 7. 接口设计

### 7.1 统一约定

后端地址默认是 `http://127.0.0.1:8080`，接口统一以 `/api` 开头。请求体使用 JSON，登录后请求头携带 `Authorization: Bearer <token>`。统一响应格式如下。

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

业务成功时 `code=200`。业务失败、参数校验失败和权限错误也通过 JSON 中的 `code` 表达，例如 400、401、403、404 和 409。当前后端统一异常处理返回 HTTP 200，前端以响应体 `code` 判断业务是否成功。

## 8. 系统实现

### 8.1 后端实现

后端入口类为 `DormRepairApplication`。控制器层按照业务模块拆分为认证、报修、用户、统计、上传和健康检查等控制器。所有业务接口返回 `ApiResponse<T>`，分页数据返回 `PageResult<T>`。业务异常通过 `BusinessException` 表达，统一异常处理器将异常转换为统一 JSON 响应。

报修业务的核心实现在 `RepairServiceImpl`。该类集中处理学生创建报修、管理员审核派单、维修人员处理、学生确认评价等流程。为了保证状态流转正确，Service 层通过 `requireStatus`、`ensureOrderStudent` 和 `ensureAssignedRepairer` 等方法校验当前状态和当前用户身份。例如，已完成工单不能再次派单，未分配给当前维修人员的工单不能开始处理，非创建学生不能确认或评价工单。

认证业务由 `AuthServiceImpl` 实现。注册时后端先检查用户名是否重复，再限制公开注册只能创建学生账号。登录时验证用户是否存在、是否被软删除、账号状态是否启用、密码是否匹配。密码使用 BCrypt 加密存储，避免明文密码进入数据库。

### 8.2 前端实现

前端以 `App.tsx` 为根组件，负责维护当前登录用户、token 校验和角色首页跳转。页面组件拆分在 `pages/` 目录中，包括登录注册、学生首页、提交报修、工单列表、进度跟踪、详情页、管理员首页、派单、用户管理、个人中心和公告页。通用交互能力下沉到 `components/`，例如 `AppShell` 负责主布局，`StatusBadge` 负责状态展示，`Timeline` 展示处理记录，`AssignModal`、`TextModal` 和 `FeedbackModal` 处理弹窗输入。

前端接口统一封装在 `api.ts`。Axios 实例设置默认 API 地址和 JSON 请求头，请求拦截器自动读取本地 token 并写入 `Authorization`。响应解包函数检查响应体 `code`，非 200 时抛出错误，由页面显示提示。该方式减少了页面层重复处理 token、错误信息和接口路径的代码。

### 8.3 页面实现与交互

系统页面遵循角色工作台思路。学生首页突出个人报修统计和快速报修入口；管理员首页突出待处理数量、趋势图和统计分布；维修人员任务页突出分配任务与处理按钮。工单详情页根据角色和状态动态显示操作按钮，例如学生在 `PENDING` 状态可撤销，在 `WAIT_CONFIRM` 状态可确认完成，在 `COMPLETED` 且未评价时可评价；管理员在 `PENDING` 状态可审核或驳回，在 `APPROVED` 或 `NEED_REASSIGN` 状态可派单；维修人员在 `ASSIGNED` 状态可开始处理，在 `PROCESSING` 状态可提交完成。

## 11. 开发难点与解决方案

### 11.1 多角色权限控制

系统涉及学生、管理员和维修人员三类角色。若只在前端隐藏按钮，用户仍可能通过直接调用接口越权操作。因此系统将权限校验放在后端：首先通过 JWT 拦截器识别当前用户，再通过 `@RequireRole` 和 Service 层校验保证业务安全。例如学生不能访问管理员用户列表，维修人员不能处理未分配给自己的工单，学生不能确认他人提交的报修单。

### 11.2 工单状态流转控制

报修业务不是简单 CRUD，工单必须按规定状态流转。若状态规则分散在前端，容易出现不同页面判断不一致。因此项目将状态流转集中在 `RepairServiceImpl` 中处理，通过枚举 `RepairStatus` 表示状态，通过 `requireStatus` 判断当前操作是否合法。例如只有 `APPROVED` 或 `NEED_REASSIGN` 状态允许派单，只有 `PROCESSING` 状态允许维修人员提交完成。

### 11.3 前后端字段一致性

前后端分离项目常见问题是字段名、枚举值和返回结构不一致。本项目通过 `types.ts` 定义前端类型，通过 `api.ts` 集中封装接口，通过后端 DTO 和 VO 明确请求与响应对象。所有状态值使用英文枚举，例如 `PENDING`、`ASSIGNED`、`COMPLETED`，前端再映射为中文文案，减少了接口层歧义。

### 11.4 数据可追溯性

报修业务需要知道每一步是谁操作、何时操作、操作内容是什么。因此系统设计了 `repair_records` 表，每次关键操作都写入处理记录。详情页通过时间线展示记录，既提升用户体验，也便于管理员追溯问题。

### 11.5 测试覆盖与回归风险

项目初期主要依赖接口测试，后来补充了单元测试。单元测试重点覆盖容易出错的规则，包括公开注册权限、角色越权、状态流转和参数校验。这样在后续修改时，可以通过 `mvn test` 快速发现业务规则是否被破坏。

项目采用“先闭环、后优化”的原则。第一优先级是保证学生提交、管理员派单、维修人员处理、学生确认评价的主流程可运行；第二优先级是补充统计、用户管理、软删除和异常处理；第三优先级是图片上传、移动端适配和文档完善。该策略有效避免了在扩展功能上投入过多时间而影响主流程交付。

## 13. 总结与展望

本项目围绕高校宿舍报修场景，完成了一套具有较完整业务闭环的小型 Web 管理系统。系统实现了学生、管理员和维修人员三类角色，覆盖注册登录、报修提交、工单查询、审核派单、维修处理、确认评价、统计分析、用户管理和图片上传等功能。后端使用 Spring Boot、Spring MVC、MyBatis 和 MySQL 构建，前端使用 React、TypeScript 和 Vite 构建，整体符合课程任务书对前后端交互、数据库持久化、权限区分、异常处理、模块化代码结构和系统可运行性的要求。

通过本次实践，团队对软件工程开发流程有了更直接的理解。需求阶段需要控制系统边界，避免功能范围过大；设计阶段需要先确定数据库结构、状态枚举和接口规范；编码阶段需要保持分层清晰，避免将业务规则分散到多个位置；测试阶段不仅要验证正常流程，还要覆盖权限、异常和非法状态流转；文档阶段需要将实现过程、测试结果和部署方式清晰记录，保证系统可以被复现和验收。

系统仍有可继续优化的空间。第一，可以进一步完善最终端到端验收记录，例如补充更细粒度的接口执行记录和部署环境说明。第二，可以扩展消息通知功能，在工单状态变化时提醒学生或维修人员。第三，可以增加 Excel 导出和操作日志，提升管理员统计和审计能力。第四，可以优化移动端体验，使学生在手机上提交报修更方便。第五，可以增加更多自动化测试，例如控制器集成测试和 Mapper 数据库测试，进一步提升回归测试质量。

总体而言，本项目完成了课程实践要求的核心目标，能够体现从需求分析到部署测试的完整软件开发过程，也为后续校园后勤信息化系统扩展提供了基础。
