# 宿舍报修管理系统详细开发流程

## 1. 开发目标

本项目需要完成一个功能完整、可正常运行、数据可持久化的小型 Web 业务系统。最终系统应支持学生提交报修、管理员审核派单、维修人员处理、学生确认评价、管理员统计管理等完整流程，并能通过课程验收和报告评审。

## 2. 总体开发路线

项目采用“先闭环、后优化”的开发方式：

1. 先完成登录注册、三类角色权限、报修主流程和数据库持久化。
2. 再补充统计、用户管理、软删除、异常处理和页面细节。
3. 最后整理测试截图、课程报告、部署说明和答辩演示材料。

核心原则：

- 每个功能必须有页面、接口、数据库记录和测试结果。
- 状态流转统一由后端校验，接口返回状态决定可操作按钮。
- 所有演示数据必须来自数据库，不能写死在代码中。
- 每天结束前记录已完成内容、未完成问题和次日计划。

## 5. 阶段二：系统设计

### 5.1 架构设计

系统采用前后端分离架构：

```text
客户端
    |
    | HTTP/JSON
    v
Spring Boot + Spring MVC + SSM REST API
    |
    | MyBatis
    v
MySQL 数据库
```

后端负责登录鉴权、角色校验、业务规则、状态流转和数据库访问。数据库负责保存用户、报修单、处理记录和评价数据。

### 5.2 数据库设计

核心表：

- `users`：用户信息、角色、账号状态、软删除时间。
- `repair_orders`：报修单基础信息、维修人员、状态、驳回原因、完成时间、软删除时间。
- `repair_records`：报修单处理记录，形成详情页时间线。
- `repair_feedbacks`：学生评价和满意度评分。

设计要点：

- 用户密码只保存加密后的哈希值。
- 报修单状态使用枚举值，避免前后端状态文案不一致。
- 派单、处理、确认等关键操作必须写入 `repair_records`。
- 删除用户和报修单优先使用软删除，避免破坏历史记录。

### 5.3 接口设计

接口统一以 `/api` 开头，统一返回：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

接口分组：

- `auth`：注册、登录、当前用户信息。
- `repairs`：报修单新增、查询、详情、修改、删除、审核、派单、处理、确认、评价。
- `users`：用户列表、维修人员列表、用户编辑、角色修改、启用禁用、软删除。
- `statistics`：总览统计、状态统计、类型统计、维修人员统计。
- `uploads`：报修图片上传，作为扩展功能。

### 5.4 输出

- 数据库表结构。
- 接口清单。
- 页面跳转关系。
- 状态流转规则。

## 6. 阶段三：后端开发

### 6.1 项目初始化

建议目录：

```text
server/
  pom.xml
  src/main/java/com/dorm/repair/
    DormRepairApplication.java
    controller/
    service/
    mapper/
    entity/
    dto/
    vo/
    config/
    interceptor/
    common/
    util/
  src/main/resources/
    application.yml
    mapper/
  uploads/
```

具体步骤：

1. 使用 Spring Initializr 或 IDE 创建 Spring Boot 项目，按 SSM 分层组织代码。
2. 添加 Spring Web、MyBatis、MySQL Driver、JWT、Lombok、Swagger / OpenAPI 等依赖。
3. 在 `application.yml` 中配置数据库连接、服务端口、JWT 密钥和可选上传目录。
4. 编写 Entity、Mapper 和 XML 映射文件，完成数据库访问。
5. 添加统一响应结构、全局异常处理和参数校验。
6. 添加 CORS 配置、日志配置、JWT 拦截器和角色权限校验。

### 6.2 登录与权限模块

开发顺序：

1. 实现用户注册接口。
2. 实现密码加密和密码校验。
3. 实现用户登录接口，返回 JWT。
4. 实现 JWT 解析拦截器或过滤器。
5. 实现角色权限拦截器。
6. 实现获取当前用户信息接口。

验收标准：

- 正确账号密码可登录。
- 错误密码返回明确提示。
- 未登录访问业务接口被拦截。
- 学生访问管理员接口被拒绝。

### 6.3 报修模块

开发顺序：

1. 学生新增报修单，默认状态为“待受理”。
2. 学生查询个人报修列表和详情。
3. 学生修改未受理报修信息。
4. 学生撤销待受理报修。
5. 管理员查询全部报修单。
6. 管理员受理或驳回报修。
7. 管理员分配维修人员。
8. 管理员软删除无效或测试报修单。

关键规则：

- 只有报修创建人可以修改或撤销自己的待受理报修。
- 已派单、处理中、待确认、已完成的报修不能被学生修改。
- 管理员驳回时必须填写驳回原因。
- 派单时必须选择维修人员。

### 6.4 维修处理模块

开发顺序：

1. 维修人员查询分配给自己的工单。
2. 维修人员将工单状态改为“处理中”。
3. 维修人员填写处理说明并提交完成。
4. 无法处理时提交原因，状态变为“需重派”。
5. 每次操作写入处理记录。

关键规则：

- 维修人员只能操作分配给自己的工单。
- 未派单工单不能开始处理。
- 已完成工单不能再次处理。

### 6.5 评价与统计模块

开发顺序：

1. 学生确认“待确认”工单。
2. 学生提交评分和评价。
3. 管理员查看统计概览。
4. 管理员查看按状态、类型、维修人员维度的统计。

关键规则：

- 只有报修创建人可以确认完成和评价。
- 评分范围为 1 到 5 分。
- 统计数据必须从数据库实时计算或查询，不能写死。

### 8.1 测试顺序

5. 管理员审核和派单接口测试。

7. 学生确认评价接口测试。
8. 管理员统计和用户管理接口测试。

### 8.2 测试检查项

- 时间格式是否统一。
- 状态枚举是否统一。
- 分页字段是否统一。
- 401 登录失效是否返回正确响应。
- 403 无权限是否返回明确提示。
- 表单校验错误是否返回具体字段和原因。

### 8.3 演示主流程

1. 使用学生账号获取 Token。
2. 调用接口提交一条宿舍报修。
3. 使用管理员账号获取 Token，受理并派单。
4. 使用维修人员账号获取 Token，开始处理并提交完成。
5. 使用学生账号获取 Token，确认完成并评价。
6. 使用管理员账号获取 Token，查看统计结果和处理记录。

## 13. 风险与应对

| 风险 | 表现 | 应对措施 |
| --- | --- | --- |
| 范围过大 | 核心流程未完成却在做扩展功能 | 先完成必做功能，图片上传和导出作为扩展 |
| 状态流转混乱 | 接口返回错误的状态 | 后端集中校验状态，客户端根据后端返回的状态决定展示 |
| 权限漏洞 | 学生能访问管理员数据 | 所有业务接口都经过 JWT 拦截器和角色权限校验 |
| 联调困难 | 前后端字段不一致 | 先固定接口文档，再开发页面和接口 |
| 报告材料不足 | 最后缺少截图和测试记录 | 开发过程中同步保存截图和测试结果 |
| 演示数据不稳定 | 演示时没有合适数据 | 准备初始化 SQL 和固定演示账号 |

- 系统能启动，后端能正常访问。

- 表单必填项、错误提示和非法访问拦截正常。
- 报修单支持新增、查询、更新和软删除。
- 页面简洁清晰，核心操作按钮明确。
- 课程报告结构完整，字数不少于 3000 中文字。
- 团队分工、工作量占比和贡献说明已填写。
- 演示账号、部署说明和初始化数据已准备完成。
