# 宿舍报修管理系统需求文档

## 1. 项目概述

### 1.1 项目名称

宿舍报修管理系统

### 1.2 项目背景

高校宿舍日常维修问题较多，例如水电故障、门窗损坏、网络异常、公共设施损坏等。传统报修方式通常依赖线下登记、电话或聊天软件沟通，容易出现信息遗漏、处理进度不透明、管理员统计困难、学生重复催办等问题。

本项目拟开发一个小型 Web 业务系统，为学生、宿舍管理员和维修人员提供统一的线上报修、派单、处理、反馈和统计入口，提升宿舍维修流程的信息化水平。

### 1.3 项目目标

1. 建立完整的宿舍报修业务闭环，支持报修提交、查询、审核、派单、处理、完成确认和评价反馈。
2. 实现普通用户、管理员、维修人员三类角色的权限区分。
3. 使用数据库保存用户、报修单、处理记录和评价数据，保证页面数据与数据库同步。
4. 提供清晰、简洁、可演示的 Web 页面，满足课程对完整性、实用性和工程规范的要求。
5. 通过模块化设计、接口设计和测试流程体现软件工程开发过程。

## 3. 需求分析

### 3.1 功能性需求

#### 3.1.1 用户注册与登录

- 系统公开注册仅面向学生用户，学生可通过学号、姓名、手机号、密码完成注册。
- 管理员账号由系统初始化产生，维修人员账号由管理员后台创建或修改用户角色产生，避免普通用户自行注册为高权限角色。
- 登录成功后根据角色进入不同首页。
- 密码需加密存储，登录失败时给出明确提示。
- 未登录用户访问业务页面时应跳转到登录页。

#### 3.1.2 学生报修

- 学生可创建报修单，填写宿舍楼、房间号、报修类型、问题描述、联系方式等信息。
- 报修类型包括水电、门窗、家具、网络、公共设施、其他。
- 当前课程基础版本以文本报修为核心，数据库预留 `image_urls` 字段；图片上传作为扩展功能，不强制实现上传接口。
- 报修单创建后默认状态为“待受理”。

#### 3.1.3 报修记录查询

- 学生可查看自己提交的全部报修记录。
- 支持按状态、报修类型、提交时间进行筛选。
- 可查看报修详情，包括处理状态、维修人员、处理记录和评价信息。

#### 3.1.4 报修审核与派单

- 管理员可查看所有报修单。
- 管理员可将待受理报修单标记为“已受理”或“已驳回”。
- 对已受理报修单，管理员可指定维修人员并生成维修任务。
- 管理员可查看各维修人员当前任务数量，辅助派单。

#### 3.1.5 维修处理

- 维修人员可查看分配给自己的工单。
- 维修人员可将工单状态更新为“处理中”。
- 维修完成后填写处理说明，并将状态更新为“待确认”。
- 若无法维修，可填写原因并提交给管理员重新处理。

#### 3.1.6 完成确认与评价

- 学生可对“待确认”的报修单进行确认完成。
- 学生可填写 1-5 分满意度评分和文字评价。
- 确认后报修单状态变为“已完成”。

#### 3.1.7 管理统计

- 管理员可查看报修总数、待受理数量、处理中数量、已完成数量。
- 支持按报修类型统计数量。
- 支持按维修人员统计完成数量和平均评价分。

#### 3.1.8 用户管理

- 管理员可查看用户列表。
- 管理员可创建维修人员账号或新增内部管理账号。
- 管理员可修改用户角色、禁用异常账号。
- 管理员可编辑用户基础信息，例如姓名、手机号和账号状态。
- 管理员可对测试账号或无效账号执行软删除，保留历史工单关联数据。
- 普通用户不可访问用户管理页面和接口。

#### 3.1.9 数据增删改查要求

为满足课程任务书中“数据可正常新增、查询、更新、删除”的要求，系统对核心数据明确提供 CRUD 能力：

| 数据对象 | 新增 | 查询 | 更新 | 删除 |
| --- | --- | --- | --- | --- |
| 用户 | 学生公开注册或管理员新增 | 用户列表、当前用户信息 | 修改角色、状态、手机号等 | 管理员软删除或禁用 |
| 报修单 | 学生提交报修 | 我的报修、全部报修、详情查询 | 审核、派单、处理、确认、评价关联更新 | 学生撤销待受理报修，管理员软删除无效或测试报修 |
| 处理记录 | 状态变更时自动生成 | 报修详情时间线查询 | 原则上不允许随意修改，必要时管理员追加更正记录 | 保留审计记录，不做物理删除 |
| 评价 | 学生确认完成后提交 | 报修详情和统计查询 | 允许学生在限定时间内修改一次评价 | 管理员可隐藏违规评价 |

### 3.2 非功能性需求

1. 可用性：页面简洁清晰，核心操作入口明显，表单具备必填校验和错误提示。
2. 安全性：后端接口必须验证登录状态和角色权限，避免越权访问。
3. 可靠性：关键数据写入数据库，不能只保存在前端页面或浏览器缓存中。
4. 可维护性：前后端代码按模块拆分，命名规范，避免大量重复代码。
5. 可测试性：核心接口具备可重复测试用例，重要流程可通过演示数据验证。
6. 性能要求：面向课程小型项目，支持几十到几百条报修数据的流畅查询即可。

## 4. 业务流程

### 4.1 报修主流程

1. 学生注册并登录系统。
2. 学生提交报修单。
3. 管理员查看待受理报修单并审核。
4. 管理员为已受理报修单分配维修人员。
5. 维修人员接收任务并处理。
6. 维修人员提交处理结果。
7. 学生确认完成并评价。
8. 管理员在统计页面查看处理情况。

### 4.2 状态流转

| 状态 | 说明 | 可操作角色 | 下一状态 |
| --- | --- | --- | --- |
| 待受理 | 学生刚提交，管理员未审核 | 学生、管理员 | 已受理、已驳回、已撤销 |
| 已受理 | 管理员确认问题有效 | 管理员 | 已派单 |
| 已派单 | 已分配维修人员 | 维修人员、管理员 | 处理中 |
| 处理中 | 维修人员正在处理 | 维修人员 | 待确认、需重派 |
| 待确认 | 维修人员已提交完成说明 | 学生 | 已完成 |
| 已完成 | 学生确认并评价 | 管理员 | 无 |
| 已驳回 | 管理员认为报修信息无效 | 学生、管理员 | 无 |
| 已撤销 | 学生主动撤销待受理报修 | 学生 | 无 |
| 需重派 | 维修人员无法处理，需要管理员重新派单 | 管理员 | 已派单 |

### 4.3 非法状态流转限制

| 当前状态 | 禁止操作 | 原因 |
| --- | --- | --- |
| 已完成 | 再次派单、再次处理 | 工单已经结束，不能重复进入维修流程 |
| 已驳回 | 维修人员接单或提交完成 | 驳回工单不会进入维修流程 |
| 待受理 | 维修人员接单 | 尚未由管理员审核和派单 |
| 处理中 | 学生撤销 | 工单已进入维修流程，不能由学生随意取消 |
| 已撤销 | 管理员受理或派单 | 用户已取消报修，工单不再处理 |
| 待确认 | 管理员重新派单 | 维修人员已提交完成说明，应由学生确认或反馈问题 |

## 5. 技术选型

### 5.1 技术栈

| 层级 | 技术 | 选择理由 |
| --- | --- | --- |
| 后端 | Java + Spring Boot + Spring MVC + SSM | 技术成熟、课程常用、分层清晰，适合开发 Web 管理系统和 REST API |
| 数据库 | MySQL 8 | 关系型数据结构清晰，适合报修单、用户、评价等业务数据 |
| 持久层 | MyBatis | SSM 常用持久层框架，便于编写 SQL、映射实体类和实现增删改查 |
| 权限认证 | JWT + Spring MVC 拦截器 | 前后端分离场景常用，便于接口鉴权和角色校验 |
| 接口文档 | Swagger / OpenAPI | 便于展示接口设计和测试接口 |
| 版本管理 | Git | 记录团队成员贡献，符合工程规范 |

### 5.2 技术方案说明

本项目采用前后端分离架构。后端使用 Java 语言，基于 Spring Boot 快速搭建项目，使用 Spring MVC 提供 RESTful API，采用 SSM 分层思想组织 Controller、Service、Mapper 和 Entity，MyBatis 负责数据库访问和对象映射。该方案技术成熟、课程适配度高，适合展示完整的软件工程开发流程。

### 5.3 部署与运行方式

- 后端通过 Spring Boot 内置 Tomcat 运行，使用 `application.yml` 或 `application.properties` 配置 MySQL 连接、JWT 密钥和服务端口。
- 后端通过 Spring Boot 内置 Tomcat 运行，生产或演示阶段可由 Nginx 反向代理后端 API。
- 图片上传属于扩展功能；若后续实现，可将上传目录配置为本地 `uploads/`，数据库仅保存文件访问地址。

## 6. 系统架构设计

### 6.1 总体架构

系统采用前后端分离架构：

- 后端负责业务逻辑、权限校验、数据处理、状态流转和接口输出。
- 数据库负责持久化存储用户、报修单、处理记录和评价数据。

```text
客户端
    |
    | HTTP/JSON
    v
Spring Boot + Spring MVC + SSM REST API
    |
    | MyBatis
    v
MySQL 数据库
```

### 6.2 后端分层

| 层级 | 职责 |
| --- | --- |
| Controller | 接收 HTTP 请求，完成参数绑定、基础校验和统一响应返回 |
| Interceptor/Filter | 处理 JWT 鉴权、角色权限、跨域、日志和异常拦截 |
| Service | 处理业务规则，例如状态流转、角色校验、派单逻辑 |
| Mapper | 封装 MyBatis 数据库访问，避免业务层直接操作 SQL 细节 |
| Entity | Java 实体类，对应用户、报修单、处理记录和评价等数据表 |
| DTO/VO | 请求参数对象和页面返回对象 |
| Config | 数据库、JWT、跨域、MyBatis、接口文档和可选上传目录等配置 |

### 6.3 后端结构

```text
server/
  pom.xml
  src/main/java/com/dorm/repair/
    DormRepairApplication.java    Spring Boot 启动类
    controller/                   Spring MVC 控制器
    service/                      业务逻辑接口与实现
    mapper/                       MyBatis Mapper 接口
    entity/                       数据库实体类
    dto/                          请求参数对象
    vo/                           响应视图对象
    config/                       数据库、跨域、接口文档等配置
    interceptor/                  JWT 鉴权和角色权限拦截
    common/                       统一响应、异常处理、常量枚举
    util/                         Token、密码加密、日期等工具
  src/main/resources/
    application.yml               项目配置
    mapper/                       MyBatis XML 映射文件
  uploads/                        可选上传目录，图片上传扩展功能使用
```

## 8. 数据库设计

### 8.5 表关系与约束设计

| 关系或字段 | 约束类型 | 说明 |
| --- | --- | --- |
| users.id -> repair_orders.student_id | 外键关系 | 一个学生可以提交多个报修单 |
| users.id -> repair_orders.repairer_id | 外键关系 | 一个维修人员可以处理多个报修单 |
| repair_orders.id -> repair_records.order_id | 外键关系 | 一个报修单可以产生多条处理记录 |
| repair_orders.id -> repair_feedbacks.order_id | 外键关系 | 一个报修单最多对应一条评价 |
| users.username | 唯一索引 | 避免学号或工号重复注册 |
| repair_orders.order_no | 唯一索引 | 保证工单号唯一，方便按工单号查询 |
| repair_orders.status | 普通索引 | 方便按状态筛选待受理、处理中、已完成等工单 |
| repair_orders.created_at | 普通索引 | 方便按提交时间排序和筛选 |
| repair_feedbacks.order_id | 唯一索引 | 保证同一个报修单只保存一条最终评价 |

删除策略说明：用户和报修单采用软删除方式实现，删除后列表默认不显示，但数据库保留 `deleted_at` 字段用于追溯历史数据。该设计既满足课程任务书中“删除”的操作要求，也保证报修记录、处理记录和评价数据的完整性。

## 9. 接口设计

### 9.1 接口约定

- 接口统一使用 `/api` 作为前缀。
- 基础版本请求与响应数据格式统一为 JSON。
- 登录成功后后端返回 JWT，客户端在后续请求头中携带 `Authorization: Bearer <token>`。
- 后端通过 Spring MVC 拦截器或过滤器完成登录校验、角色校验、跨域处理和异常拦截。
- 接口返回结构统一如下：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

- 分页列表建议统一使用 `page`、`page_size`、`total`、`items` 字段。
- 业务错误由后端返回明确提示，例如“无权限访问”“状态不允许变更”“记录不存在”。

## 16. 实现风险与范围控制

### 16.1 主流程优先

本项目功能点较多，开发时应优先保证以下主流程完整可运行：

```text
登录 -> 学生提交报修 -> 管理员审核派单 -> 维修人员处理 -> 学生确认评价 -> 管理员查看统计
```

只要该主流程稳定运行，就能覆盖课程任务书中的用户入口、基础业务功能、数据持久化、权限区分、异常处理和前后端交互要求。

### 16.2 功能优先级

| 优先级 | 功能 | 处理策略 |
| --- | --- | --- |
| P0 | 登录注册、角色权限、报修提交、审核派单、维修处理、确认评价 | 必须完成，作为验收演示主线 |
| P1 | 基础统计、用户启用禁用、修改角色、报修单软删除 | 主流程完成后实现，用于提升系统完整性 |
| P2 | 图片上传、消息通知、Excel 导出、移动端适配、超时提醒 | 扩展功能，时间允许再实现 |

### 16.3 技术选型说明

任务书未强制技术栈，允许使用成熟稳定、可落地的 Web 技术。选择 SSM、Spring Boot 和 Spring MVC 是因为该技术路线课程适配度高、资料丰富、分层清晰，配合 MyBatis 和 MySQL 可以规范完成课程要求的增删改查、权限认证和业务流程实现。

### 16.4 软删除说明

本系统中的删除采用软删除方式实现。删除后数据在页面列表中默认不显示，但数据库保留 `deleted_at` 字段用于追溯历史数据，既满足课程任务书中的“删除”操作要求，也保证用户、报修单、处理记录和评价之间的历史关联不被破坏。

### 17.2 固定演示脚本

1. 使用 API 测试工具（如 Postman 或 curl），调用 `POST /api/auth/login` 接口，使用学生账号 `student001` 登录，获取 JWT Token。
2. 携带学生 Token，调用 `POST /api/repairs` 接口，提交一条宿舍报修，例如”1号楼 602 宿舍灯不亮”。
3. 调用 `GET /api/repairs/my` 接口，查看刚提交的工单，确认状态为”待受理”。
4. 调用 `POST /api/auth/login` 接口，使用管理员账号 `admin001` 登录，获取管理员 JWT Token。
5. 携带管理员 Token，依次调用 `POST /api/repairs/{id}/approve` 受理工单，再调用 `POST /api/repairs/{id}/assign` 将工单分配给维修人员 `repair001`。
6. 调用 `POST /api/auth/login` 接口，使用维修人员账号 `repair001` 登录，获取维修人员 JWT Token。
7. 携带维修人员 Token，调用 `POST /api/repairs/{id}/start` 开始处理，再调用 `POST /api/repairs/{id}/finish` 填写处理说明并提交完成。
8. 携带学生 Token，调用 `POST /api/repairs/{id}/confirm` 确认维修完成，再调用 `POST /api/repairs/{id}/feedback` 填写评分和评价。
9. 携带管理员 Token，调用 `GET /api/statistics/overview` 接口，查看总工单数、处理中数量、已完成数量等统计数据变化。
10. 携带学生 Token，尝试调用 `GET /api/users` 接口（管理员接口），系统应返回无权限错误，证明权限拦截有效。

### 17.3 演示检查点

- 页面刷新后数据仍存在，证明数据已经持久化到数据库。
- 每次状态变化后详情页均出现处理记录，证明流程可追溯。
- 不同角色看到的菜单和按钮不同，证明权限控制生效。
- 已完成工单不能再次派单，证明后端状态流转校验有效。
